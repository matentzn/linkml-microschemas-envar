Environmental-exposure microschema · first community draft

A unified, tiered
metadata layer for
environmental exposures.

Every pipeline that joins an environmental value to a patient record already carries metadata. Each does it differently, and none agree on what "complete" means. EnVar proposes one representation and a community-reviewed set of requirement tiers.

PLAN.md + SPEC.md TMAX + PM2.5 a technical walkthrough
The problem

Geo-joining a value drops the context that makes it interpretable

When a gridded environmental value (a daily maximum temperature, a PM2.5 concentration) is attached to a patient record, the value travels but its derivation does not. A downstream analyst cannot reproduce or correctly interpret it from the data alone.

Lost — provenance

which product, which version, which model and its known biases

Lost — geometry

grid resolution, CRS, extraction rule, geography type

Lost — semantics

day-boundary convention, aggregation method, units

A concrete failure

Two studies, same county and summer, opposite results

Study A — Daymet

Significant association

Extreme-heat days linked to emergency admissions.

Study B — PRISM

Null result

Same source family of temperature data. No association.

Both analyses are competent. The choices that produced the difference were never recorded in either dataset.

Why they disagreed

Two undocumented choices

Definition of "a day"

A: local midnight → local midnight (Daymet). B: 24 h ending 12:00 GMT (PRISM). The same afternoon lands on different calendar days, so the two "daily maxima" are different series.

Definition of "extreme"

A: 95th percentile of 1971–2000. B: 95th percentile of 2000–2019. In a warming climate these are different thresholds and flag different days.

Day-boundary convention is the most-omitted field in the heat-epidemiology literature. At least seven heat-wave definitions are in active use; none are convertible after the fact.

The current state — not a vacuum

Every pipeline already carries metadata

The problem is not that metadata is missing. It is that each system records it in its own representation, at a different granularity, with different gaps — and no shared definition of what "enough" is.

Example · OMOP

The external_exposure table

external_exposure (proposed OMOP extension)
person_id             # who
location_id           # where (aggregated geography)
exposure_concept_id   # what exposure
exposure_start_date
exposure_end_date
value_as_number       # the value
unit_concept_id       # the unit

Carries the value, a concept id, a period and a unit. This is a single-value model.

  • no grid resolution or CRS
  • no extraction or linkage rule
  • no model type, biases, or uncertainty
  • many exposures have no concept id at all

The table is a proposed CDM extension, not yet ratified. EnVar's provenance_id is what its exposure_source_value would hold.

Example · GAIA / gaiaCatalog

Three metadata files per dataset

discovery

meta_dcat_*.json

DCAT record for catalogue search.

processing

meta_etl_*.json

custom ETL instructions for ingest.

linked data

meta_json-ld_*.json

Schema.org JSON-LD for the catalogue.

GAIA describes metadata at the dataset level ("PM2.5, CMAQ, US, 2000–2020, 12 km"). EnVar describes it at the variable-value level ("this value is an annual mean in µg/m³ at tract level, population-weighted from 12 km CMAQ"). They are complementary — and LinkML emits the JSON-LD GAIA consumes.

Example · ORNL C-HER

The one system that already enforces a bar

What it is

ORNL's Centralized Health and Exposomic Resource (Hanson). 30+ spatially indexed exposomic datasets — air quality, climate, water, built environment, social context — unified with population health data.

Preprint: arXiv:2511.03750. Internal data formats are not publicly documented — itself an argument for a shared representation.

The precedent EnVar generalises
  • Registration-time completeness gate — a resource is not "metadata-complete" until its required fields are filled.
  • EnVar's tiered checker is the same move, made variable-level and tier-aware.

Working relationship: advisory; C-HER datasets are EnVar's ETL validation set; monthly standards reviews align C-HER, GAIA, and EnVar.

The common gap

No shared representation, no shared bar

  • Different representations — relational table, JSON-LD, file attributes, free text, container output.
  • Different granularities — dataset vs variable vs value vs row.
  • Different, undeclared gaps — each drops different fields; none state what they omit.
  • No agreed requirement set — nothing says which fields a record must have to be reproducible.
The proposal

One representation + tiered requirements + a checker

1 — representation

A variable-level sidecar

A LinkML schema covering identity, space, time, source, model, linkage, provenance. Exports to JSON Schema, SHACL, OWL.

2 — requirements

Community-reviewed tiers

Core / Recommended / Optional, annotated on each slot — a shared, machine-readable bar for "complete".

3 — measurement

A completeness checker

Scores any sidecar against the tiers. Reads tiers from the schema, so it can't disagree with the spec.

A sidecar carries no PHI. It stops at the hook that links to the patient data and never holds clinical data itself.

The design goal — it governs every element decision

Reproducible from raw data plus metadata alone, without access to the pipeline.

An element is in the schema only if its absence would block reproduction, correct interpretation, or discovery. The tiers (next) record how much its absence blocks.

The question is not what can be captured, but what must be

Three requirement tiers

Core

The minimum

Without it the record is meaningless. Missing → BLOCKING.

Recommended

The ideal

Reproducible in principle without it; real reproduction needs it. Missing → ⚠ warning.

Optional

The maximum

Enriches discovery and cross-linking, not reproduction. Missing → noted.

The tier assignment in SPEC.md is a strawman authored to be challenged. The workshop moves elements between tiers.

A fourth tier

Conditionally-Core

Optional in general; mandatory once a stated context holds.

Context: derived heat metric

equation_variant, indoor_outdoor, heat_wave_threshold_definition, percentile reference period become required.

Context: buffer / station / ensemble

buffer radius, max station distance, ensemble member count become required when those strategies are used.

Encoded as LinkML rules: when the context predicate is true, the associated slots are required.

A required pattern

Null with reason

Every nullable element, when null, must carry a reason from a controlled vocabulary. A blank is a validation error; a null-with-reason is information.

blank — rejected
target_concept_id:
null + reason — valid
target_concept_id: null
target_concept_id_missing_reason: not_provided_by_source

Vocabulary: not_provided_by_source · available_but_not_extracted · upstream_data_not_propagated · under_investigation · not_applicable.

Testing the claim

Can you reproduce it from the sidecar alone?

The sidecar specifies enough to re-run and byte-verify
  • which product + version (version, doi)
  • the exact tool + image (tool_version, container_image_digest)
  • the exact invocation (run_arguments)
  • extraction, day-boundary, linkage parameters
  • verify the result by input/output_file_sha256
It does not guarantee — three external conditions must hold
  • the raw source is still retrievable
  • the container image still resolves from its digest
  • the computation is deterministic (no unseeded randomness)

Honest claim: the sidecar makes the analysis reconstructable and byte-verifiable in principle — not push-button reproducible. It records the conditions reproduction depends on; it cannot enforce that they still hold.

The completeness checker

A score read straight from the schema

Input a sidecar; it reports present, missing, and a readiness score. Tiers come from the schema annotations, so the score cannot disagree with the spec.

$ uv run envar-check EnvironmentalExposureRecord-tmax_core_missing.yaml

Element detailCore

day_boundary_convention

Daymet uses local midnight; PRISM uses 24 h ending 12:00 GMT. A "daily maximum" depends on the window chosen, and cross-product comparison is sensitive to it.

local_midnightutc_midnightending_1200_gmtsolar_noon_centeredobservation_dependent

Proposed as Core deliberately, as the field whose omission breaks the most studies. SPEC flags it for the community to downgrade if they disagree.

Element detail · grid → personCore

extraction_method + linkage_strategy

The largest single lever in exposure assignment: the same point gets a different value depending on the rule.

extraction_methodbehaviour
nearest_celltake the cell the point falls in
inverse_distance_weighted_4_nearest_cellsdistance-weight the 4 nearest cells
area_weighted_polygon_meanaverage over a polygon (tract, ZCTA)
population_weighted_meanweight cells by where people live

LinkageMethod is the "linkage descriptor" the GECC/EIRENE forum names as the central gap. Buffer radius and max-station-distance are Conditionally-Core where those strategies apply.

Element detail · error structureCore

exposure_model_type

A measured value, an interpolated value, and an ML-predicted value have different error structures and cannot be pooled naively. This field separates them.

direct_measurementspatial_interpolationreanalysisstatistical_blendchemical_transport_modelensemble_machine_learningsingle_machine_learningequation_derivedsatellite_retrieval

Daymet = interpolation; Di et al. PM2.5 = ensemble ML (CV R² 0.86); ACAG = satellite retrieval.

Element detail · derived heat metricsConditionally-Core

equation_variant

For wet-bulb globe temperature the approximations diverge materially from the reference model — beyond 2–3 °C in hot-humid conditions. For a derived metric, the equation used must be recorded.

WBGT

liljegren_2008 · acsm_simplified · bernard_simplified

Heat Index

rothfusz_1990_nws · steadman_1979 — undefined below ~80 °F / 40 % RH

UTCI

brode_2012_polynomial

Element detail · what counts as extremeConditionally-Core

Percentile reference period

"95th percentile" over 1971–2000 vs 2000–2019 is a different threshold in a warming climate; it flags different days and shifts the mortality estimate.

Required for heat-wave flags

At least seven definitions in use, none convertible after the fact. Capture threshold definition, min consecutive days, reference period start/end.

Scope & window

local_tract · county · climate_region · national; seasonal window warm_season_may_sep.

Generalisation check

PM2.5 — same frame, different stresses

PM2.5 reuses the entire frame but exercises different elements than heat: ML/satellite model types instead of interpolation, an annual-vs-daily averaging distinction, a different linkage story.

Model type

Non-interpolation

monitor → ensemble ML → satellite retrieval.

Averaging window

Daily vs annual

Acute (~2-day) and chronic (annual) PM2.5 are distinct exposures. Made explicit.

Concept status

gap → proposed

PM2.5 sits in the OHDSI Exposome vocabulary effort; no single canonical id yet.

How the element set was derived

Re-derived from source requirements

  • Re-derive from the naked requirements — 70+ slots across 13 sections, each with a stated source-of-truth.
  • Ground in evidence — every Core and high-controversy element carries a PMID / DOI / standard URL.
  • Add PM2.5 — extend with what a pollutant needs that heat does not exercise.
  • Reconcile against the prototype — element by element, where the set matches, adds, drops, or renames the existing 13-module schema.

New work for v0.2: tier annotations, reference annotations, and the top-level naming decision — not a structural rebuild.

Community session

Five judgements per element

  • 1 · Core? mandatory, or the record is meaningless
  • 2 · Necessary for reproducibility? Recommended
  • 3 · Nice-to-have? Optional
  • 4 · Well described? is the definition clear and correct
  • 5 · Missing? what belongs here that isn't

Output: a tier-reclassification list + a missing-element list, folded into schema v0.2 and a revised SPEC.
