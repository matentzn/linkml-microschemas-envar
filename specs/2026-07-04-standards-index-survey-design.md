# Standards-index survey design — cataloguing metadata approaches for environmental-exposure data

**Date:** 2026-07-04
**Status:** Implemented (2026-07-04)
**Author:** Nico Matentzoglu

## 1. Problem

EnVar has a good answer to *"isn't this the same as X?"* for five named standards
(`docs/related-approaches.md`, `specs/2026-07-03-standard-coverage-design.md`).
What it does not have is a **wide-angle map of the field**: the full set of approaches that
record *metadata about environmental-exposure data* — optionally paired with health outcomes —
for the purpose of **reuse, reproducibility, and meta-analysis**.

The space is murky and large (see `todo.md` for the landscape and a first, admittedly partial,
list of candidate standards). Two kinds of thing belong in the index and are easy to conflate:

- **General standards** — specs *intended* to be reusable and interoperable across producers.
- **Structured-but-unstandardised metadata** — per-pipeline documentative formats (DeGAUSS and
  Amadeus sidecars, project-local YAML) that are structured but never aimed to be general.

We want both, kept apart by an explicit class tag. We also want the messy candidate list in
`todo.md` **triaged, not dropped**: every candidate gets an in-scope / peripheral / out-of-scope
verdict with a one-line reason.

Prior material exists in `~/ws/notes/niehs_standards/surveys/` (`projects_inventory.md`, the
`study-env-vars-health-outcomes/REPORT.md`, the issue-003 landscape report). It catalogues
*projects*, not *metadata standards*, and it is not assumed complete. This survey runs a fresh
net and reconciles against that material at the end.

## 2. Scope

### In scope — indexed as entries

Anything that **defines or records metadata fields** about environmental-exposure data for reuse.
Two tagged classes:

- **`general_standard`** — e.g. DCAT, ISO 19115 / 19139, ISO 19156 (Observations & Measurements),
  schema.org / science-on-schema.org, STAC, RO-Crate, Croissant, DDI, W3C PROV, CDIF, CODATA
  Essential Variables, OMOP GIS / Exposome CDM extensions, DataCite, Frictionless Data Package.
  (Illustrative, not exhaustive — the survey discovers the real set.)
- **`pipeline_sidecar`** — DeGAUSS sidecar, Amadeus sidecar, and other structured-but-per-pipeline
  documentative formats.

### Out of scope — referenced, not indexed

- **Controlled vocabularies / ontologies** (ENVO, SWEET, CF standard names, QUDT, AGROVOC, GCMD,
  SVO, and the bulk of the `todo.md` list). These appear only in an entry's `vocabularies_used`
  field, never as their own entry.
- **Projects and data platforms** (CDC Environmental Public Health Tracking, exposome cohorts,
  TISLab internal projects). These appear only in the reconciliation appendix or an entry's
  `notes`, never as their own entry.
- **Coverage scoring and crosswalks.** Whether a standard carries a given EnVar slot is the
  separate concern of `2026-07-03-standard-coverage-design.md`. This survey answers *"what is
  out there and is it relevant?"*, not *"how well does it cover EnVar?"*.

## 3. Deliverables

Three artifacts, with the registry as the single source of truth.

1. **Registry** — `docs/standards-index/standards.yaml`. One entry per standard, structured
   (see §4). Hand-curated from the survey output; the thing that gets queried and maintained.
2. **Rendered index** — `docs/standards-index/index.md`, generated from the registry. A narrative
   intro that draws the `general_standard` vs `pipeline_sidecar` line and reuses the
   unit-of-description / primary-purpose / layer axes from `docs/related-approaches.md`, followed
   by tables grouped by class and relevance.
3. **Reconciliation appendix** — a section of the rendered `index.md` mapping each entry to
   `~/ws/notes/niehs_standards/surveys/projects_inventory.md`
   and `study-env-vars-health-outcomes/REPORT.md`: **new** (surfaced by this survey) vs
   **already-catalogued** (and where).

### Registry format decision

The registry is modelled with a **small standalone LinkML schema**
(`docs/standards-index/standards_index.schema.yaml`), kept **out of the `linkml_microschemas_envar`
umbrella** so it cannot affect `EnvironmentalExposureRecord` or the `gen-project` outputs of the
envar schema. This dogfoods the toolchain and gives free validation of the registry. **Fallback:**
if wiring the standalone schema into the build proves fiddly, the registry degrades to plain
documented YAML validated by a JSON Schema — the entry shape in §4 is identical either way.

The rendered index and reconciliation appendix are generated by a small Python module,
`standards_index.py`, a sibling to the existing `checker.py` / `coverage.py`, behind a new
`just standards-index` recipe declared in `project.justfile`.

## 4. Registry entry model

Each entry is keyed by a short `id` slug and carries:

### Core fields (required for in-scope entries)

| Field | Meaning |
|---|---|
| `id` | short kebab/snake slug |
| `name` | short display name |
| `class` | `general_standard` \| `pipeline_sidecar` |
| `spec_url` | canonical spec / homepage |
| `governance` | body + maturity (ISO / W3C / OGC / RDA / CODATA / community / single-project) |
| `unit_of_description` | `dataset` \| `variable_type` \| `geocoded_row` \| `produced_value` \| `file` |
| `primary_purpose` | `find` \| `define` \| `compute` \| `reproduce_pool` \| `cite` |
| `layer` | `discovery` \| `conceptual` \| `instance` \| `linkage` |
| `carries_derivation_metadata` | `yes` \| `partial` \| `no` |
| `relevance` | `in_scope` \| `peripheral` \| `out_of_scope` |
| `relevance_reason` | one line justifying the relevance verdict |
| `envar_relation` | one line: which EnVar concern(s) it overlaps; complementary vs competing |
| `evidence` | one or more verified source URLs |

`unit_of_description`, `primary_purpose`, and `layer` deliberately reuse the vocabulary already
established in `docs/related-approaches.md` so the index and the narrative page stay commensurable.

### Optional fields

| Field | Meaning |
|---|---|
| `full_name` | expanded name / what the acronym stands for |
| `domain_specificity` | `env_health_specific` \| `general_purpose` |
| `vocabularies_used` | controlled vocabularies / ontologies the standard leans on (links the `todo.md` vocab list without making them entries) |
| `reconciliation` | `new` \| `already_catalogued`, with a pointer into the niehs inventory / study REPORT |
| `notes` | short free text |

### Relevance triage rule

Every candidate — including every line of the `todo.md` list that a research agent touches —
must end with a `relevance` verdict and a `relevance_reason`. `out_of_scope` entries **are kept**
in the registry (or a clearly-marked triage tail) so the survey is auditable: a reader can see
*"we looked at GeoNames and ruled it out because it is a gazetteer, not an exposure-metadata
standard."* Silent omission is not allowed.

## 5. Execution — multi-agent deep research

The survey runs as a multi-agent fan-out (a Workflow, matching the chosen multi-agent approach),
built and executed from the implementation plan — **not** during brainstorming.

**Sweep (multi-modal, agents blind to each other).** Fan out research agents on orthogonal angles
so no single search framing bounds coverage:

- by **standards body** — ISO / TC 211, OGC, W3C, RDA, CODATA, DDI Alliance, DataCite;
- by **domain** — earth/climate data metadata, clinical/health data standards, geospatial
  observation models, provenance, FAIR / research-data packaging;
- by **ecosystem** — OMOP / OHDSI, exposome cohorts, NIEHS / EPA / CDC, LinkML / OBO.

**Seed set.** The `todo.md` candidate list, the five standards in `related-approaches.md`, and the
standards/vocabularies columns of `projects_inventory.md`.

**Converge.**

1. **Dedupe / merge** candidate lists across all sweep agents.
2. **Adversarially verify** each candidate — a skeptic pass answering: does it exist, is it a
   *metadata* standard (not a vocabulary, not a project), and is it in scope? Default to
   `out_of_scope` when uncertain, with the reason recorded.
3. **Classify** survivors into the §4 model (`class`, axes, relevance, `envar_relation`).
4. **Reconcile** every entry against the niehs inventory and the study REPORT.
5. **Completeness critic** — a final agent asking "what angle, body, or ecosystem was not swept?";
   its findings seed one more sweep round. Loop until a round adds nothing new.

Every in-scope entry must carry at least one **verified** source URL before it lands in the
registry.

## 6. Success criteria

- The registry validates (LinkML) or parses (fallback YAML) cleanly.
- Every candidate the survey touches — including each `todo.md` line — has a `relevance` verdict
  and a one-line reason; nothing is silently dropped.
- `general_standard` and `pipeline_sidecar` entries are distinguishable and the rendered index
  draws that line in prose.
- `just standards-index` regenerates the rendered index and reconciliation appendix from the
  registry alone.
- The reconciliation appendix shows, per entry, new vs already-in-niehs-inventory.
- Every in-scope entry has a verified source citation.

## 7. Out of scope (this pass)

- No coverage scoring or crosswalks against EnVar slots (that is
  `2026-07-03-standard-coverage-design.md`).
- No controlled vocabularies / ontologies or projects / platforms as first-class entries.
- No changes to `linkml_microschemas_envar` schema modules or the profile.
- No attempt to make the registry schema part of the envar umbrella or the `gen-project` build.
