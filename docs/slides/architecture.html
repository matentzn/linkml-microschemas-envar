The one idea to hold on to

The schema set describes one produced value series.

Everything is metadata about a value (or series) a pipeline already emitted.

Design goal: the series is reproducible from raw data + this metadata alone — without access to the pipeline.

The components,
one by one.

Eleven small schemas — one class, one concern each. For every value you saw in the CSV.

Component 01 / 11 core

VariableIdentity

What physical quantity is being measured, in what units, bound to which vocabularies.

standard_name:   air_temperature      # CF standard name
units_ucum:      Cel
value_data_type: continuous_numeric
concept_status:  gap                 # vocab status (existing|proposed|gap)

Answers: "what is this number?"

Component 02 / 11 core

DataLayout

The map between the sidecar and the companion data file — which columns carry the values, subjects, and dates.

table_orientation: wide            # wide (column per variable) | long (tidy)
value_column:      tmax
subject_column:    subject_id
time_column:       date

Answers: "where in the file are the numbers this sidecar describes?"

Component 03 / 11 core

SpatialReference

The native grid, its coordinate system, and how a grid cell became a point value.

native_spatial_resolution_m: 1000
crs:                         EPSG:4326
extraction_method:           inverse_distance_weighted_4_nearest_cells
target_geography_type:       point_residence

Answers: "where — and how was a grid turned into one location?"

Component 04 / 11 core

TemporalReference

The time window, how it was aggregated, and — critically — which day-boundary convention.

temporal_resolution:         daily
temporal_aggregation_method: maximum
day_boundary_convention:     local_midnight   # the most-omitted field
calendar:                    gregorian

Answers: "over what window, under which definition of a day?"

Component 05 / 11 core

SourceDataset

The identity of the upstream product the values originate from.

source_dataset_name:    "Daymet V4 Daily Surface Weather"
source_dataset_version: "V4 R1"          # version changes the values
source_dataset_doi:     10.3334/ORNLDAAC/2129
source_license_spdx:    public-domain-us-gov

Answers: "which product, which version?"

Component 06 / 11 core

ExposureModel

How the values were produced — model class, inputs, and known biases.

exposure_model_type:         spatial_interpolation
exposure_model_paper_doi:    10.3334/ORNLDAAC/2129
exposure_model_known_biases: ["warm-season warm bias…"]
exposure_model_cross_validation_r2: null  # a model has different error than a measurement

Answers: "measured, interpolated, or modelled — and how well?"

Component 07 / 11 recommended

Uncertainty

Per-value and whole-model uncertainty, quality flags, and how gaps were filled.

per_value_uncertainty_field_name: tmax_stderr
per_value_uncertainty_type:       standard_error
missing_data_handling_method:     spatiotemporal_interpolation
data_completeness_pct:            100

Answers: "how sure are we, and what was filled in?"

Component 08 / 11 core

LinkageMethod

How a place-based value was attached to a patient location — the join itself.

linkage_strategy:               point_extraction_at_residence
geocoding_precision_propagated: range
geocoding_score_propagated:     0.95
address_period_alignment:       address_history_from_emr

Answers: "how did the value reach a person?"

Component 09 / 11 core + chain: recommended

ToolRun + ProvenanceChain

The exact invocation that produced the value, plus the ordered upstream runs (W3C PROV).

tool_name: daymet   tool_version: 1.0.0
container_image_digest: sha256:a8b3c2…   # strongest repro anchor
input_file_sha256 / output_file_sha256
provenance_chain: geocoder 3.3.0 → daymet 1.0.0

Answers: "exactly which run, and what came before it?"

Component 10 / 11 conditionally-core

DerivedHeatMetric

Only for heat indices: the equation, indoor/outdoor, and percentile baseline. Omitted for plain Tmax.

heat_metric_family:       wbgt_outdoor
equation_variant:         liljegren_2008   # variants differ by 2–3 °C
indoor_outdoor:           outdoor
heat_wave_threshold_definition: percentile_local

Answers: "for a derived metric — which formula and thresholds?"

Component 11 / 11 recommended + deposit: optional

HealthLayerLinkage + DepositMetadata

The forward hooks: how it links into OMOP, and what it needs to travel as a FAIR deposit.

health_layer_target: omop_external_exposure   health_layer_link_field: exposure_source_value  # carries provenance_id
deposit_doi:          10.5281/zenodo.9999999
deposit_license_spdx: CC-BY-4.0
dcat_distribution_url: https://zenodo.org/…/tmax.parquet

Answers: "how does it hook into the health layer and get published?"

The picture — how the 14 microschemas relate

High-level architecture

The Profile is an external import. The 14 EnVar files = common + 11 components + record + examples; border colour marks each component's tier.

The composite, up close

It does exactly two things

1 · realises

The profile anatomy — under readable names

variable_identity · spatial_reference · temporal_reference · exposure_model map to the profile's observation_type · location · temporality · methodology via implements; subject is verbatim; observation_result is not bound — the values stay in the data file.

2 · extends

The envar-specific slots

Concerns with no profile equivalent — data_layout, source_dataset, linkage_method, tool_run, … — added directly.

instantiates: MicroschemaDefinition, and every component is inlined — the sidecar is one self-contained graph, no identifiers to resolve.

Four conventions run through every component

Know these and the rest is mechanical

naming

Readable names at the top

Validated keys are the domain names (variable_identity, …); each maps to its profile slot via implements. Resolved 2026-07 — still open to workshop challenge.

requirements

Tiers as annotations

Every slot carries tier: core | recommended | optional — the schema is the requirement spec.

missingness

Null with reason

Every nullable slot pairs with a *_missing_reason. A blank is invalid; a null-with-reason is information.

composition

Inline, identifier-free

All component objects are inlined — one graph, nothing to dereference.

Tiers in practice · 1 / 3 core · the minimum

A Core sidecar — only the mandatory slots

validates · minimally reproducible

temporal_reference: { temporal_resolution: daily, temporal_aggregation_method: maximum, day_boundary_convention: local_midnight }

Miss any one of these and the completeness checker blocks. This is the floor.

Tiers in practice · 2 / 3 recommended · full reproducibility

Add Recommended — what real reproduction needs

everything in Core, plus…
source_dataset:   + source_dataset_doi: 10.3334/ORNLDAAC/2129   + source_license_spdx: public-domain-us-gov
exposure_model:   + exposure_model_paper_doi: …   + exposure_model_known_biases: ["warm-season warm bias…"]
linkage_method:   + geocoding_precision_propagated: range   + geocoding_score_propagated: 0.95

uncertainty:      + data_completeness_pct: 100

Core + Recommended → someone else can reproduce it end-to-end.

The shape in five sentences

  • One sidecar describes one produced value series; the numbers stay in the data file.
  • Read it left to right — value → linkage → run → model → source → space/time → quantity.
  • Each step is a component: eleven small modules, one concern each.
  • They stack into a six-layer cake; the composite realises the profile anatomy under readable names and extends with envar slots, all inlined.
  • Four patterns: readable-names-mapped-to-the-profile, tiers-as-annotations, null-with-reason, inline composition.
