"Isn't this just the same as X?"

Each neighbour overlaps with part of EnVar. But each operates at a different granularity, or for a different purpose.

This deck draws those lines so the overlaps stop being confusing.

First, the thing we're comparing against

An EnVar sidecar is an instance-level provenance record.

One document per (variable, run), describing how a specific environmental value was produced — in enough detail to reproduce it from raw data plus the sidecar alone, and to check whether two values may legitimately be pooled.

The word that does the work

It describes one instance — not a catalogue

it does NOT

Catalogue datasets

That's discovery — a different job.

it does NOT

Define which variables matter

That's a prioritisation framework.

it does NOT

Geocode addresses

That's a computation tool.

It records the methodological identity of a value that has already been produced.

How to tell any two of these apart

Two axes separate everything

Axis 1

Unit of description

What is one record about? A whole dataset, a variable type, a geocoded row, or one produced value?

Axis 2

Purpose

Do you want to find it, define it, compute it, or reproduce / pool it?

Keep these two questions in mind for every system that follows.

01
GAIA catalog files

The dataset catalogue

What it is

A catalogue, three files per dataset

The GAIA toolchain (OHDSI GIS Working Group) catalogues place-based exposure datasets. Each one carries:

FileStandardRole
meta_dcat_*.jsonDCATdiscovery record for catalogue search
meta_etl_*.jsoncustomETL instructions for ingest
meta_json-ld_*.jsonSchema.org JSON-LDlinked-data description
Granularity — the crate vs the item

One card describes a whole product

A GAIA card says

"PM2.5, CMAQ, US, 2000–2020, 12 km."

Built to help you find a dataset — the label on the whole crate.

An EnVar sidecar says

"this value is an annual mean in µg/m³ at tract level, population-weighted from the 12 km grid."

Describes one item taken out of the crate and assigned to a person.

So how do they differ?

The run-time fields can't live on a dataset card

extraction_method, day_boundary_convention, geocoding_precision, container_image_digest are properties of the extraction run — they don't exist at the dataset level, so they can't sit on a GAIA card.

Complementary, not competing. Use GAIA to discover the product, EnVar to describe each value you extract. And because EnVar is authored in LinkML, it can emit the same Schema.org JSON-LD GAIA consumes — so the detail flows up into the catalogue.

02
DeGAUSS

The geocoding tool

What it is

A privacy-preserving geocoder, in a container

  • Geocodes a patient address and appends geomarker columns — including the extracted environmental value.
  • Runs as versioned Docker containers inside the institution — no PHI leaves the site.
  • Records the geocoding precision and match score, and the exact image used.

Granularity: per geocoded row — metadata about this linkage event.

So how do they differ?

A producer, not a description of what it did

Not alternatives — the EnVar "ideal" TMAX example is a DeGAUSS → Daymet run. DeGAUSS is strong where it acts and silent on the rest:

AxisDeGAUSS recordsEnVar adds
Linkageprecision, score, imagenormalised LinkageMethod slots
Source(implicit in container)version, DOI, license
Model—class, biases, CV R²
Temporal—day-boundary, aggregation
Tiering—Core/Recommended/Optional + score

DeGAUSS guarantees reproducible computation; EnVar guarantees the choices it made are written down in a standard, scoreable form.

03
Amadeus

The data-access layer

What it is

An R package for accessing environmental data

Amadeus (NIEHS, Messier group) pulls and processes large environmental datasets — GridMET, NARR, MERRA-2, MODIS, EPA AQS, and ~20 more. When it downloads, it can attach an acknowledgement string and file hashes.

Granularity: per downloaded file — it answers "where did these bytes come from, and how do I cite them."

So how do they differ? · same value, two representations

Amadeus is the "before" picture

what Amadeus emits today
acknowledgement: "Data provided by the University of Idaho; GridMET via Amadeus."
input_file_sha256: 3a1b9c…
output_file_sha256: 7f2e4d…
# EnVar then demands the rest, or records why it's missing:
source_dataset_version, day_boundary_convention, extraction_method, exposure_model_type …

Useful, and genuinely thin. EnVar standardises and extends what an Amadeus run should emit — and reports what is still missing.

04
CODATA Essential Variables

The concept layer

What it is

Curated sets of variables worth observing

GCOS

ECV

Essential Climate Variables

GOOS

EOV

Essential Ocean Variables

GEO BON

EBV

Essential Biodiversity Variables

CODATA generalised these into a family of frameworks — and noticed the interesting part isn't the variable lists, it's the data-description semantics underneath: what lets you decide whether two measurements of "the same" variable may be combined.

The key structure · DDI-CDI variable cascade

Three layers of "variable"

LayerMeansExample
ConceptualVariablewhat is measured & whyair temperature as a heat stressor
RepresentedVariablevalue domain + unitsair_temperature, °C
InstanceVariablethis column, this file, this runthe tmax value from Daymet V4 R1

Essential Variables live at the top two layers. EnVar lives at the bottom one — the InstanceVariable detail an EV definition deliberately doesn't carry.

The relationship

Same move, opposite ends of one cascade

Essential Variables / CDIFEnVar sidecar
Entry layerconceptual — "what is the claim?"instance — "how was this number made?"
Directiontop-down (hypothesis → evidence)bottom-up (bytes → identity)
Answers"do we mean the same variable?""may I pool these two values?"

The shared thesis is commensurability. 43.91 °C (Daymet) vs 42.65 °C (GridMET) for the same patient-day is a commensurability failure wearing the costume of two equal numbers — answered at the layer where the method differences actually live.

The takeaway on CODATA

Speak the vocabulary, don't adopt the toolchain

EnVar is not a competitor to CDIF — it's the missing instance-layer detail that makes a CDIF document safe to pool. A CDIF document can say "temp_c is air temperature"; it cannot say "…extracted by inverse-distance-weighting four cells, on a local-midnight day boundary, from Daymet V4 R1." That second sentence is EnVar.

Cheap, high-leverage alignment: expose an I-ADOPT projection of each variable, and label which cascade layer each field sits on (CF/UCUM → RepresentedVariable; provenance → InstanceVariable).

Check yourself

Can you answer each in one line?

  • vs GAIA? dataset card (find it) vs one value's provenance (reproduce it) — complementary via JSON-LD.
  • vs DeGAUSS? a producer of values vs a standard description of what the producer did.
  • vs Amadeus? acknowledgement + hashes (the "before") vs the full, scoreable record.
  • vs Essential Variables? the concept layer ("same variable?") vs the instance layer ("pool these values?").

One line to keep: EnVar fills the instance layer the others leave underspecified.
